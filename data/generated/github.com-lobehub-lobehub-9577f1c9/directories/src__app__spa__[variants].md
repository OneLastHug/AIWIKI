# 目录：src/app/spa/[variants]

## 它负责什么

`src/app/spa/[variants]` 是 Next.js App Router 下的 SPA HTML 服务入口。它不是 React SPA 自身的路由配置，而是负责在服务端响应 `/spa/{variants}/...` 这类请求，返回一份已经注入运行时配置、SEO 元信息和前端静态资源引用的 HTML。

这个目录的核心职责可以概括为：

1. 根据 URL 中的 `variants` 解析当前页面变体，例如语言 `locale`、是否移动端 `isMobile`。
2. 选择合适的 SPA HTML 模板：开发环境从本地 Vite dev server 获取，生产环境使用构建产物模板。
3. 在 HTML 中注入 `window.__SERVER_CONFIG__`，把服务端全局配置、feature flags、analytics 配置、客户端环境变量等传给浏览器端 SPA。
4. 注入 SEO meta 信息，包括 title、description、Open Graph、Twitter card 等。
5. 返回最终 HTML，让浏览器继续加载真正的 SPA JS/CSS 资源。

从代码形态看，它是“Next.js 服务端 HTML 外壳”和“Vite/React SPA 前端应用”之间的桥接层。

## 关键组成

目标目录实际结构很小：

```text
src/app/spa/[variants]
└── [[...path]]
    ├── route.ts
    ├── mobileHtmlTemplate.source.ts
    └── spaHtmlTemplates.d.ts
```

`[[...path]]` 是 Next.js 的可选 catch-all 路由段。它表示 `variants` 后面的任意路径都可以被同一个 `route.ts` 接住。例如根据当前片段推断，类似下面的路径都会落到这里：

```text
/spa/{variants}
/spa/{variants}/chat
/spa/{variants}/settings
/spa/{variants}/任意前端路由
```

这里的 `path` 参数在 `GET` 里没有直接使用，说明后续路径主要是为了让浏览器刷新 SPA 内部页面时仍然拿到同一份 HTML，真正的页面匹配交给客户端的 `react-router-dom`。

`route.ts` 是核心文件，主要导出两个 Next.js 约定函数：

```ts
export function generateStaticParams()
export async function GET(...)
```

`generateStaticParams()` 负责生成静态变体参数。它使用：

```ts
RouteVariants.serializeVariants({ isMobile, locale })
```

为 `en-US`、`zh-CN` 以及移动端/桌面端组合生成 `variants`。如果当前是桌面端构建，即 `isDesktop` 为真，则只生成桌面选项；否则同时生成 mobile 和 desktop 两类选项。

`GET()` 是请求处理入口。它从 `params` 中取出 `variants`，再通过：

```ts
RouteVariants.deserializeVariants(variants)
```

解析出：

```ts
locale
isMobile
```

随后构造 `spaConfig`，注入 HTML，并返回 `Response`。

`mobileHtmlTemplate.source.ts` 是移动端 HTML 模板源文件，文件头注明：

```ts
// Auto-generated by scripts/mobileSpaWorkflow
// Do not edit manually
```

它导出：

```ts
export const mobileHtmlTemplate = "..."
```

模板中包含：

- `<!doctype html>` 基础 HTML 壳。
- `<!--SEO_META-->` 占位符。
- 初始化主题的脚本。
- 初始化语言、`document.documentElement.lang` 和 RTL 方向的脚本。
- `window.__SERVER_CONFIG__ = undefined; /* SERVER_CONFIG */` 占位符。
- 指向 `[URL已移除] 的移动端 JS/CSS 资源。
- `<!--ANALYTICS_SCRIPTS-->` 占位符。

`spaHtmlTemplates.d.ts` 只声明生产模板模块的类型：

```ts
export declare const desktopHtmlTemplate: string;
export declare const mobileHtmlTemplate: string;
```

实际 `./spaHtmlTemplates` 文件不在当前片段中出现，结合 `.d.ts` 和 `mobileHtmlTemplate.source.ts` 的注释，可以判断它大概率是构建流程生成的产物，供生产环境动态导入。

## 上下游关系

上游入口是 Next.js App Router。因为该目录位于：

```text
src/app/spa/[variants]/[[...path]]/route.ts
```

所以 Next.js 会把它注册为 `/spa/[variants]/[[...path]]` 的 Route Handler。浏览器请求这个路径时，会进入 `GET()`。

它依赖的上游配置和服务主要包括：

```ts
getServerGlobalConfig()
getServerFeatureFlagsValue()
translation('metadata', locale)
analyticsEnv
appEnv
fileEnv
pythonEnv
RouteVariants
serializeForHtml()
```

这些依赖分别提供：

- `getServerGlobalConfig()`：服务端全局配置。
- `getServerFeatureFlagsValue()`：服务端 feature flags。
- `translation('metadata', locale)`：按语言读取 metadata 翻译。
- `analyticsEnv`：Google Analytics、Plausible、Umami、Clarity、PostHog、X Ads、Vercel Analytics 等配置。
- `appEnv`：如 `MARKET_BASE_URL`。
- `fileEnv`：如 `NEXT_PUBLIC_S3_FILE_PATH`。
- `pythonEnv`：如 Pyodide 相关资源地址。
- `RouteVariants`：负责把 `locale/isMobile` 编码进 URL 变体字符串，也负责反解。
- `serializeForHtml()`：把对象安全序列化进 HTML，避免直接拼接 JSON 带来的转义风险。

下游是浏览器里的 SPA。`route.ts` 返回的 HTML 会创建：

```html
<div id="root" style="height: 100%"></div>
```

然后加载对应的前端 JS。前端启动后，会读取：

```ts
window.__SERVER_CONFIG__
```

来获得服务端注入的配置，再由 SPA 自己的路由系统接管页面渲染。

因此这个目录不是业务页面本身，而是业务页面的“HTML 启动器”。

## 运行/调用流程

完整流程可以按一次页面请求理解：

1. 用户访问 `/spa/{variants}/...`。
2. Next.js 命中 `src/app/spa/[variants]/[[...path]]/route.ts`。
3. `GET()` 从 `params` 里读取 `variants`。
4. `RouteVariants.deserializeVariants(variants)` 解析出当前语言和是否移动端。
5. 服务端读取全局配置：

```ts
const serverConfig = await getServerGlobalConfig();
const featureFlags = getServerFeatureFlagsValue();
const analyticsConfig = buildAnalyticsConfig();
const clientEnv = buildClientEnv();
```

6. 组合成 `spaConfig`：

```ts
const spaConfig: SPAServerConfig = {
  analyticsConfig,
  clientEnv,
  config: serverConfig,
  featureFlags,
  isMobile,
};
```

7. 根据环境选择模板。

开发环境：

```ts
const res = await fetch('[URL已移除]');
const html = await res.text();
return rewriteViteAssetUrls(html);
```

也就是说开发时会从 Vite dev server 取 HTML。随后 `rewriteViteAssetUrls()` 会把 HTML 里以 `/` 开头的脚本和样式地址改成 `[URL已移除] module import 与 Worker 加载问题。这是为了让线上或代理环境中的 HTML 能正确加载本地 Vite 资源。

生产环境：

```ts
const { desktopHtmlTemplate, mobileHtmlTemplate } = await import('./spaHtmlTemplates');
return isMobile ? mobileHtmlTemplate : desktopHtmlTemplate;
```

也就是直接使用构建好的模板字符串。

8. 替换服务端配置占位符：

```ts
window.__SERVER_CONFIG__ = undefined; /* SERVER_CONFIG */
```

替换为真实的序列化配置：

```ts
window.__SERVER_CONFIG__ = { ... };
```

9. 调用 `buildSeoMeta(locale)` 生成 SEO 标签，并替换：

```html
<!--SEO_META-->
```

10. 当前代码把：

```html
<!--ANALYTICS_SCRIPTS-->
```

替换为空字符串。根据当前片段推断，analytics 配置并不是在这里直接插入脚本，而是通过 `window.__SERVER_CONFIG__.analyticsConfig` 交给前端运行时处理。

11. 返回 HTML：

```ts
return new Response(html, {
  headers: {
    'Cache-Control': 'no-cache',
    'content-type': 'text/html; charset=utf-8',
  },
});
```

这里设置 `no-cache`，说明这份 HTML 是运行时配置敏感的，不适合被浏览器或中间层长期缓存。静态 JS/CSS 资源本身则通过模板中的资源 URL 单独缓存。

## 小白阅读顺序

建议按下面顺序读：

1. 先看目录结构，理解 `src/app/spa/[variants]/[[...path]]/route.ts` 是一个 Next.js Route Handler，而不是 React 页面组件。
2. 读 `GET()`，因为它是请求入口，先弄清楚一次请求如何变成一份 HTML。
3. 再读 `getTemplate(isMobile)`，理解开发环境和生产环境的模板来源不同。
4. 接着读 `buildAnalyticsConfig()`，看服务端如何把各种统计平台配置收集进统一对象。
5. 再读 `buildClientEnv()`，理解哪些环境变量会暴露给浏览器端 SPA。
6. 然后读 `buildSeoMeta(locale)`，理解 metadata 翻译和 SEO 标签是如何注入的。
7. 最后读 `mobileHtmlTemplate.source.ts`，把模板里的占位符和 `route.ts` 中的 replace 逻辑对应起来。

特别要注意两个占位符：

```html
<!--SEO_META-->
```

由 `buildSeoMeta(locale)` 替换。

```ts
window.__SERVER_CONFIG__ = undefined; /* SERVER_CONFIG */
```

由 `serializeForHtml(spaConfig)` 替换。

只要理解这两个替换点，就能明白这个目录的主要价值：它把构建期生成的 SPA HTML，变成带有服务端运行时上下文的最终 HTML。

## 常见误区

第一个误区：把 `[variants]` 当成普通业务路由参数。

这里的 `variants` 不是某个页面 ID，而是编码后的运行变体，至少包含 `locale` 和 `isMobile`。真正的业务页面路径在 `[[...path]]` 中被兜底接住，后续交给浏览器端 SPA 路由处理。

第二个误区：以为 `[[...path]]` 里的 `path` 没用就是冗余。

它虽然在 `GET()` 里没有读取，但它的作用是让任意 SPA 子路径刷新时都能返回同一份 HTML。例如用户在前端路由深层页面刷新，如果没有 catch-all，很可能会 404。

第三个误区：以为这个目录负责渲染 React 页面。

它只返回 HTML 壳，不直接渲染业务组件。React 组件、页面布局、客户端路由主要在 `src/spa`、`src/routes`、`src/features` 等目录中。

第四个误区：以为生产环境也会访问 `localhost:9876`。

`localhost:9876` 只在 `process.env.NODE_ENV === 'development'` 时使用。生产环境会动态导入 `./spaHtmlTemplates`，读取构建产物里的 `desktopHtmlTemplate` 或 `mobileHtmlTemplate`。

第五个误区：手动修改 `mobileHtmlTemplate.source.ts`。

文件头明确说明它由 `scripts/mobileSpaWorkflow` 自动生成，不应手动编辑。需要改模板来源或生成逻辑时，应去找对应生成脚本或构建流程，而不是直接改这个文件。

第六个误区：忽略 `serializeForHtml()` 的安全意义。

服务端配置最终会被写进 `<script>` 中。如果直接 `JSON.stringify()` 拼接，可能引入 HTML/script 转义问题。这里使用 `serializeForHtml(spaConfig)`，说明项目有专门的安全序列化工具。

第七个误区：认为 analytics 脚本在服务端直接注入。

当前 `route.ts` 构造了 `analyticsConfig`，但最后把 `<!--ANALYTICS_SCRIPTS-->` 替换为空。根据当前片段推断，统计脚本更可能由前端根据 `window.__SERVER_CONFIG__.analyticsConfig` 初始化，而不是由这个 Route Handler 直接输出。

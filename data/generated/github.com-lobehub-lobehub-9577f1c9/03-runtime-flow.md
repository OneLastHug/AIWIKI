# 运行与流转

启动链路可以从 `scripts/devStartupSequence.mts` 读起。根据当前文件推断，开发模式会同时拉起 Next.js 和 Vite：前者负责 `src/app` 中的 API、HTML 模板和认证页，后者负责 `src/spa` 编译出的 SPA 资源。这个脚本会先加载 `.env` 系列文件，再启动 `next dev -p 3010` 与 `bun run dev:spa`，然后等待 Next 就绪并预热根路径。`vite.config.ts` 里打印 `Debug Proxy` 的逻辑则说明，SPA 本地开发时会通过一个代理 URL，把本地 Vite 服务器挂到线上环境里调试。

浏览器端启动链路比较固定。`src/app/spa/[variants]/[[...path]]/route.ts` 根据路由变量解析出 `locale` 和 `isMobile`，再从 `src/server/globalConfig/index.ts` 取服务端全局配置、从 `src/server/featureFlags` 取开关、从 `src/envs/*` 组装客户端环境，最后把这些内容序列化到 HTML 模板里。模板来源有两个分支：开发模式下会从 `[URL已移除] 拉 Vite 页面并重写静态资源 URL；生产模式下会直接使用打包后的 HTML 模板。这个流程说明 SPA 不是完全静态的，它依赖服务端注入的运行时状态。

前端挂载后，`src/spa/entry.web.tsx`、`src/spa/entry.desktop.tsx`、`src/spa/entry.mobile.tsx` 或 `src/spa/entry.popup.tsx` 会把对应的路由树交给 `createAppRouter`。`src/utils/router.tsx` 在这里负责组装 `SPAGlobalProvider`、`BusinessGlobalProvider`、错误边界、导航注册器和预取逻辑。`src/layout/SPAGlobalProvider/index.tsx` 会读取 `window.__SERVER_CONFIG__`，决定语言、主题、全局 Store 初始化、弹窗宿主和上传上下文；这意味着首屏可用状态不是靠纯前端猜测，而是靠服务器先发给浏览器。

请求流最容易理解的是 tRPC。`src/app/(backend)/trpc/lambda/[trpc]/route.ts` 和 `src/app/(backend)/trpc/async/[trpc]/route.ts` 都会先把 `NextRequest` 克隆成适合 `fetchRequestHandler` 的请求，再分别挂到 `lambdaRouter` 与 `asyncRouter`。`src/libs/trpc/lambda/context.ts` 负责构建上下文，里面会按顺序处理调试用户、API Key、OIDC、市场 token、客户端 IP、trace context 等信息。`src/server/routers/lambda/index.ts` 把 `agent`、`session`、`topic`、`user`、`message`、`market`、`subscription` 等子路由挂起来，所以客户端服务如 `src/services/session/index.ts` 只需要调用 `lambdaClient.session.*` 即可。

Agent 运行链路是整个仓库最重的业务流。根据 `src/server/services/agentRuntime/AgentRuntimeService.ts`、`src/server/modules/AgentRuntime/factory.ts`、`src/server/services/queue/*`、`src/server/agent-hono/index.ts` 和 `src/app/(backend)/api/agent/stream/route.ts`，可以推断它大致是这样工作的：客户端发起执行请求后，`/api/agent` 入口交给 Hono app，Hono 再按路径分派到 `execAgent`、`runStep`、`toolResult`、`finalizeAbandoned`、`gatewayStart`、`gatewayCron`、`messengerWebhook` 等处理器；执行时 `AgentRuntimeService` 会创建 Agent 实例、协调状态机、记录 trace、执行工具、调度队列；如果是本地模式，`LocalQueueServiceImpl` 用 `setTimeout` 续跑下一步，如果是队列模式，`QStashQueueServiceImpl` 会把 step 发布到 QStash。SSE 端点则通过 `createStreamEventManager()` 订阅运行事件，把 `agent_runtime_end` 之前的实时步骤推回客户端。

数据库流与运行流是耦合但分层清楚的。`src/server/services/agentRuntime/AgentRuntimeService.ts` 会直接接收 `LobeChatDatabase` 和 `userId`，再配合 `packages/database/src/models/message.ts`、`packages/database/src/models/session.ts`、`packages/database/src/models/task.ts`、`packages/database/src/models/topic.ts` 等模型进行读写。`packages/database/src/core/db-adaptor.ts` 采用懒加载缓存数据库实例，避免模块导入时就初始化连接。由此可以看出，业务层不应该直接绕过 service 去碰 schema，而应该通过 model/repository/service 的组合访问数据。

桌面端和 CLI 的流转相对独立，但仍复用同一套共享包。`apps/desktop/src/main/index.ts` 会创建 Electron `App` 并调用 `bootstrap()`；`apps/desktop/electron.vite.config.ts` 负责 main/preload/renderer 三段构建；`src/services/electron/*` 与 `packages/electron-*` 则为桌面桥接提供能力。CLI 侧由 `apps/cli/src/index.ts` 进入 `createProgram()`，再注册 `agent`、`group`、`message`、`model`、`plugin`、`provider`、`topic`、`user` 等命令。CLI 会通过 `apps/cli/src/api/client.ts`、`apps/cli/src/tools/*` 和 tRPC/HTTP client 与同一后端协作，因此它不是独立后端，而是另一种调用壳。

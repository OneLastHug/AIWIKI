# 文件：src/app/spa/[variants]/[[...path]]/route.ts

## 它负责什么

这个文件是 Next.js App Router 下的一个服务端路由处理器，负责为 LobeHub 的 SPA 应用返回最终的 HTML 壳页面。

它对应的路由形态是：

```text
/spa/[variants]/[[...path]]
```

其中：

- `[variants]` 是编码后的路由变体信息，主要包含 `locale` 和 `isMobile`。
- `[[...path]]` 是可选 catch-all 路径，用来承接 SPA 内部的任意前端路径。
- 这个文件只导出 `GET`，因此它处理浏览器访问 SPA 页面时的 HTML 响应。

它的核心职责可以概括为四件事：

1. 根据 `variants` 判断当前请求是移动端还是桌面端，以及使用什么语言。
2. 获取 SPA HTML 模板，开发环境从 Vite dev server 读取，生产环境从构建产物模板读取。
3. 在 HTML 中注入服务端配置、feature flags、analytics 配置、客户端环境变量和 SEO meta。
4. 返回 `text/html` 响应，让前端 SPA 在浏览器里接管后续路由。

换句话说，这个文件不是普通 API，也不是 React 页面组件，而是“SPA HTML 外壳生成器”。

## 关键组成

### `generateStaticParams()`

```ts
export function generateStaticParams()
```

它为 Next.js 静态生成阶段提供 `[variants]` 参数。

逻辑是：

- 如果当前是桌面端构建环境，`mobileOptions` 只包含 `false`。
- 否则同时生成移动端和桌面端两类变体：`true`、`false`。
- 静态语言固定为：
  - `en-US`
  - `zh-CN`
- 每组 `{ locale, isMobile }` 会通过 `RouteVariants.serializeVariants()` 序列化成字符串，作为动态路由参数 `variants`。

因此它会生成类似这样的路由参数结构：

```ts
{
  variants: RouteVariants.serializeVariants({ isMobile, locale }),
}
```

这里的重点是：`variants` 不是随便拼出来的字符串，而是由 `RouteVariants` 统一编码和解码。

### `isDev` 与 `VITE_DEV_ORIGIN`

```ts
const isDev = process.env.NODE_ENV === 'development';
const VITE_DEV_ORIGIN = '[URL已移除]';
```

这两个常量决定 HTML 模板来源。

开发环境下，服务端不会使用静态构建好的 SPA 模板，而是直接请求本地 Vite dev server：

```ts
fetch('[URL已移除]')
```

这配合仓库里的 `dev:spa` 模式使用。Vite 负责前端 HMR，Next.js 这个 route 负责把 HTML 套进线上或本地服务端环境。

### `rewriteViteAssetUrls(html)`

```ts
async function rewriteViteAssetUrls(html: string): Promise<string>
```

这个函数只服务于开发环境。

开发环境中，从 Vite dev server 拿到的 HTML 往往包含相对路径资源，例如：

```html
<script src="/src/entry.web.tsx"></script>
<link href="/@vite/client" />
```

但是当前 HTML 是由 Next.js route 返回的，浏览器解析 `/xxx` 时会默认请求当前 Next 服务的 origin，而不是 Vite 的 `[URL已移除]

所以这个函数会用 `linkedom` 解析 HTML，并把部分资源路径改成完整的 Vite 地址：

- 所有 `script[src]` 中以 `/` 开头的路径，改成 `[URL已移除]
- 所有 `link[href]` 中以 `/` 开头的路径，改成 `[URL已移除]
- 对内联 `script type="module"` 中形如 `from "/@..."` 的 import 也做替换
- 额外插入一段 `Worker` patch，处理跨 origin worker 脚本加载问题

这段逻辑的目的不是业务功能，而是让“Next 返回 HTML + Vite 提供资源”的开发模式正常工作。

### `getTemplate(isMobile)`

```ts
async function getTemplate(isMobile: boolean): Promise<string>
```

这是 HTML 模板入口。

开发环境：

```ts
const res = await fetch(VITE_DEV_ORIGIN);
const html = await res.text();
return rewriteViteAssetUrls(html);
```

生产环境：

```ts
const { desktopHtmlTemplate, mobileHtmlTemplate } = await import('./spaHtmlTemplates');

return isMobile ? mobileHtmlTemplate : desktopHtmlTemplate;
```

根据邻近文件可见，`spaHtmlTemplates.d.ts` 声明了：

```ts
export declare const desktopHtmlTemplate: string;
export declare const mobileHtmlTemplate: string;
```

同目录下还有 `mobileHtmlTemplate.source.ts`，文件头标明：

```ts
// Auto-generated by scripts/mobileSpaWorkflow
// Do not edit manually
```

这说明生产模板很可能由构建或脚本流程生成，而不是手写维护。

模板中有两个关键占位符：

```html
<!--SEO_META-->
window.__SERVER_CONFIG__ = undefined; /* SERVER_CONFIG */
<!--ANALYTICS_SCRIPTS-->
```

当前 route 会替换前两个，并把 analytics 脚本占位符替换为空字符串。

### `buildAnalyticsConfig()`

```ts
function buildAnalyticsConfig(): AnalyticsConfig
```

它从 `analyticsEnv` 和部分 `process.env` 中收集前端需要知道的分析平台配置。

支持的配置包括：

- Google Analytics
- Plausible
- Umami
- Microsoft Clarity
- PostHog
- X Ads
- React Scan
- Vercel Analytics
- Desktop Umami 配置

每个配置都有开关和必要字段判断。例如 Google Analytics 只有在同时满足：

```ts
analyticsEnv.ENABLE_GOOGLE_ANALYTICS
analyticsEnv.GOOGLE_ANALYTICS_MEASUREMENT_ID
```

时才会写入：

```ts
config.google = {
  measurementId: analyticsEnv.GOOGLE_ANALYTICS_MEASUREMENT_ID,
};
```

这避免把未启用或不完整的 analytics 配置暴露给前端。

### `buildClientEnv()`

```ts
function buildClientEnv(): SPAClientEnv
```

它收集 SPA 运行时需要的一些公开客户端环境信息：

```ts
return {
  marketBaseUrl: appEnv.MARKET_BASE_URL,
  pyodideIndexUrl: pythonEnv.NEXT_PUBLIC_PYODIDE_INDEX_URL,
  pyodidePipIndexUrl: pythonEnv.NEXT_PUBLIC_PYODIDE_PIP_INDEX_URL,
  s3FilePath: fileEnv.NEXT_PUBLIC_S3_FILE_PATH,
};
```

这些字段类型来自 `src/types/spaServerConfig.ts` 中的 `SPAClientEnv`：

```ts
export interface SPAClientEnv {
  marketBaseUrl?: string;
  pyodideIndexUrl?: string;
  pyodidePipIndexUrl?: string;
  s3FilePath?: string;
}
```

它们最终会进入 `window.__SERVER_CONFIG__`，供浏览器端 SPA 使用。

### `buildSeoMeta(locale)`

```ts
async function buildSeoMeta(locale: string): Promise<string>
```

它根据语言生成页面 SEO 和社交分享 meta。

主要依赖：

- `translation('metadata', locale)` 获取服务端翻译函数 `t`
- `BRANDING_NAME`
- `ORG_NAME`
- `OFFICIAL_URL`
- `OG_URL`
- `isCustomORG`

它生成的内容包括：

- `<title>`
- description
- Open Graph meta
- Twitter card meta

标题和描述来自 metadata 命名空间：

```ts
const title = t('chat.title', { appName: BRANDING_NAME });
const description = t('chat.description', { appName: BRANDING_NAME });
```

这里说明 SPA HTML 虽然是壳页面，但仍然会在服务端注入基础 SEO 信息。

### `GET()`

```ts
export async function GET(
  _request: Request,
  { params }: { params: Promise<{ path?: string[]; variants: string }> },
)
```

这是实际处理 HTTP GET 请求的入口。

它做的事情是：

1. 从 `params` 里取出 `variants`。
2. 用 `RouteVariants.deserializeVariants(variants)` 解出：
   - `locale`
   - `isMobile`
3. 获取服务端全局配置：
   - `getServerGlobalConfig()`
4. 获取 feature flags：
   - `getServerFeatureFlagsValue()`
5. 构造 analytics 配置：
   - `buildAnalyticsConfig()`
6. 构造客户端环境变量：
   - `buildClientEnv()`
7. 合成 `spaConfig`：
   ```ts
   const spaConfig: SPAServerConfig = {
     analyticsConfig,
     clientEnv,
     config: serverConfig,
     featureFlags,
     isMobile,
   };
   ```
8. 根据 `isMobile` 获取 HTML 模板。
9. 把模板中的：
   ```js
   window.__SERVER_CONFIG__ = undefined; /* SERVER_CONFIG */
   ```
   替换成真实配置：
   ```ts
   window.__SERVER_CONFIG__ = ${serializeForHtml(spaConfig)};
   ```
10. 生成并替换 SEO meta。
11. 清空 analytics script 占位符。
12. 返回 HTML 响应。

响应头是：

```ts
{
  'Cache-Control': 'no-cache',
  'content-type': 'text/html; charset=utf-8',
}
```

`no-cache` 表示浏览器或中间缓存每次使用前都需要重新验证，适合这种带服务端运行时配置注入的 HTML 壳。

## 上下游关系

### 上游：谁会调用它

这个 route 的上游是浏览器访问 SPA 页面时的 HTTP GET 请求。

路径形态大致是：

```text
/spa/{variants}
/spa/{variants}/some/client/side/path
```

其中 `[[...path]]` 是可选 catch-all，所以 SPA 内部很多路径都可以落到这个 route 上。注意 `GET` 函数签名里虽然声明了：

```ts
path?: string[]
```

但当前实现没有使用 `path`。这意味着对于不同的 SPA 子路径，服务端返回的是同一个 HTML 壳，具体页面由浏览器端 React Router 再决定。

另一个上游是 Next.js 构建或静态生成机制，它会调用：

```ts
generateStaticParams()
```

来得到预生成的 `[variants]` 参数。

### 中游：它依赖哪些模块

这个文件依赖几类模块。

第一类是品牌、URL 和版本常量：

```ts
BRANDING_NAME
ORG_NAME
OG_URL
OFFICIAL_URL
isCustomORG
isDesktop
```

它们影响 SEO、Twitter site 和静态参数生成。

第二类是环境配置：

```ts
analyticsEnv
appEnv
fileEnv
pythonEnv
```

它们被整理成 `analyticsConfig` 和 `clientEnv`，再注入给前端。

第三类是服务端配置和功能开关：

```ts
getServerGlobalConfig()
getServerFeatureFlagsValue()
```

它们组成 `SPAServerConfig` 的主要业务配置部分。

第四类是国际化与安全序列化：

```ts
translation()
serializeForHtml()
```

`translation()` 用来按语言生成 SEO 文案；`serializeForHtml()` 用来把服务端对象安全地放入 HTML `<script>` 中，避免直接 `JSON.stringify` 可能带来的 HTML 注入问题。

第五类是路由变体工具：

```ts
RouteVariants
```

从 `src/utils/server/routeVariants.ts` 可见，本地的 `RouteVariants` 是对 `@lobechat/desktop-bridge` 中 `RouteVariants` 的扩展封装，并增加了适配 Next props 的 helper：

```ts
getVariantsFromProps()
getIsMobile()
getLocale()
```

当前文件直接使用它的：

```ts
serializeVariants()
deserializeVariants()
```

### 下游：它把数据交给谁

它最终把数据交给浏览器里的 SPA。

关键注入点是：

```js
window.__SERVER_CONFIG__
```

注入对象类型是 `SPAServerConfig`，结构来自 `src/types/spaServerConfig.ts`：

```ts
export interface SPAServerConfig {
  analyticsConfig: AnalyticsConfig;
  clientEnv: SPAClientEnv;
  config: GlobalServerConfig;
  featureFlags: Partial<IFeatureFlags>;
  isMobile: boolean;
}
```

因此前端启动后可以从全局变量读取：

- 服务端全局配置
- 功能开关
- analytics 配置
- 客户端环境变量
- 当前是否移动端

HTML 下游还包括实际加载的 SPA bundle：

- 开发环境加载 `[URL已移除] 提供的 Vite 资源。
- 生产环境加载模板中已经写好的构建产物资源，例如同目录生成模板里的 `[URL已移除]

## 运行/调用流程

1. 用户访问某个 SPA 地址，例如：
   ```text
   /spa/{variants}/chat
   ```

2. Next.js 根据文件路径匹配到：
   ```text
   src/app/spa/[variants]/[[...path]]/route.ts
   ```

3. `GET()` 被调用，拿到动态参数：
   ```ts
   params.variants
   ```

4. `RouteVariants.deserializeVariants(variants)` 解码出：
   ```ts
   locale
   isMobile
   ```

5. 服务端并行意义上的数据准备开始，但代码中是顺序写法：
   ```ts
   const serverConfig = await getServerGlobalConfig();
   const featureFlags = getServerFeatureFlagsValue();
   const analyticsConfig = buildAnalyticsConfig();
   const clientEnv = buildClientEnv();
   ```

6. 组装 SPA 服务端配置：
   ```ts
   const spaConfig: SPAServerConfig = {
     analyticsConfig,
     clientEnv,
     config: serverConfig,
     featureFlags,
     isMobile,
   };
   ```

7. 调用 `getTemplate(isMobile)` 获取 HTML：
   - 开发环境：请求 Vite dev server，再重写资源地址。
   - 生产环境：读取 `desktopHtmlTemplate` 或 `mobileHtmlTemplate`。

8. 替换服务端配置占位符：
   ```ts
   html = html.replace(
     /window\.__SERVER_CONFIG__\s*=\s*undefined;\s*\/\*\s*SERVER_CONFIG\s*\*\//,
     `window.__SERVER_CONFIG__ = ${serializeForHtml(spaConfig)};`,
   );
   ```

9. 调用 `buildSeoMeta(locale)` 生成 SEO 标签，并替换：
   ```html
   <!--SEO_META-->
   ```

10. 清空 analytics script 占位符：
    ```ts
    html = html.replace('<!--ANALYTICS_SCRIPTS-->', '');
    ```

11. 返回 HTML 响应：
    ```ts
    return new Response(html, {
      headers: {
        'Cache-Control': 'no-cache',
        'content-type': 'text/html; charset=utf-8',
      },
    });
    ```

12. 浏览器收到 HTML 后加载 JS/CSS bundle，前端 React SPA 接管页面渲染和后续路由。

## 小白阅读顺序

1. 先看文件路径：
   ```text
   src/app/spa/[variants]/[[...path]]/route.ts
   ```
   这能帮助你理解它是 Next.js App Router 的 route handler，而不是 React 组件。

2. 再看最底部的 `GET()`。
   这是主流程入口，读懂它就能知道这个文件整体在做什么。

3. 接着看 `RouteVariants.deserializeVariants(variants)`。
   这里决定当前请求的语言和移动端/桌面端形态。可以顺手看：
   ```text
   src/utils/server/routeVariants.ts
   ```

4. 然后看 `getTemplate(isMobile)`。
   这是理解开发环境和生产环境差异的关键：
   - development：从 Vite dev server 取 HTML。
   - production：从 `spaHtmlTemplates` 读取模板字符串。

5. 再看 HTML 替换逻辑。
   重点关注两个占位符：
   ```html
   window.__SERVER_CONFIG__ = undefined; /* SERVER_CONFIG */
   <!--SEO_META-->
   ```

6. 然后读 `buildAnalyticsConfig()` 和 `buildClientEnv()`。
   这两个函数告诉你有哪些环境变量会被暴露给浏览器端 SPA。

7. 最后读 `buildSeoMeta(locale)`。
   它和主业务启动关系不大，但解释了为什么 SPA 壳页面也能有基础 SEO 信息。

## 常见误区

1. 误以为这是一个页面组件。

   这个文件没有导出 React component，也没有 JSX。它导出的是 `GET()`，本质是 HTTP route handler，负责返回 HTML 字符串。

2. 误以为 `[[...path]]` 会在服务端决定具体页面。

   当前代码没有使用 `path` 参数。服务端对不同 SPA 子路径基本返回同一个 HTML 壳，具体页面由浏览器端 SPA 路由处理。

3. 误以为生产环境也会请求 Vite。

   只有 `NODE_ENV === 'development'` 时才会请求：
   ```text
   [URL已移除]
   ```
   生产环境使用构建生成的 `desktopHtmlTemplate` 或 `mobileHtmlTemplate`。

4. 误以为 `window.__SERVER_CONFIG__` 是前端自己生成的。

   它是在这个 route 里由服务端注入的。前端拿到页面时，HTML 中已经包含真实配置。

5. 误以为可以直接用 `JSON.stringify(spaConfig)` 注入 HTML。

   当前代码使用：
   ```ts
   serializeForHtml(spaConfig)
   ```
   这通常是为了避免对象内容破坏 `<script>` 上下文或引入 HTML 注入风险。学习这里时应注意“把 JSON 放进 HTML”不是简单字符串拼接问题。

6. 误以为 analytics 脚本是在这里直接生成的。

   这个文件只构造 `analyticsConfig` 并注入到 `window.__SERVER_CONFIG__`。模板里的：
   ```html
   <!--ANALYTICS_SCRIPTS-->
   ```
   在当前实现中被替换为空字符串。实际 analytics 初始化更可能由前端 SPA 根据配置完成；这是根据当前片段推断，依据是本文件没有插入任何具体 analytics `<script>` 标签。

7. 误以为 `generateStaticParams()` 覆盖所有语言。

   它这里只静态生成：
   ```ts
   ['en-US', 'zh-CN']
   ```
   这不等于系统只支持这两种语言。它只是这个 route 的静态参数生成策略。

8. 误以为移动端和桌面端只是 CSS 响应式差异。

   这里会根据 `isMobile` 选择不同 HTML 模板：
   ```ts
   isMobile ? mobileHtmlTemplate : desktopHtmlTemplate
   ```
   也就是说移动端和桌面端可能加载不同的 SPA 构建入口和资源。

9. 误以为 `Cache-Control: no-cache` 表示完全不缓存。

   `no-cache` 的含义更接近“可以存储，但使用前必须重新验证”。这里这样设置，是因为 HTML 中注入了服务端配置、feature flags 和环境相关信息，不适合长期强缓存。

10. 误以为同目录的 `mobileHtmlTemplate.source.ts` 应该手动编辑。

   文件头已经说明：
   ```ts
   Auto-generated by scripts/mobileSpaWorkflow
   Do not edit manually
   ```
   这类模板源文件通常由构建流程生成，手动改动容易被下次生成覆盖。
